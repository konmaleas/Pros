Previous Architectural Problem (Now SOLVED)

The Fundamental Issue: Circular dependency between TitleSearch and FrequencyBasedFilter

    FrequencyBasedFilter needed frequency data for smart thresholds
    But frequency data came from TitleSearch.title_analysis() which ran on already-filtered text
    Major Redundancy: Both classes did frequency counting independently

NEW PIPELINE ARCHITECTURE DESIGNED
Stage 1A: TextExtractor ✅

    Pure PDF → Raw text extraction using your existing PdfToText.pdf_extract_text()
    No processing, single responsibility

Stage 1B: TextProcessor ✅

    Raw text → Clean line-split text using your exact clear_file_() logic
    Character normalization only, no content filtering

Stage 2: FrequencyAnalyzer ✅

    Clean lines → Complete frequency analysis
    Consolidates ALL frequency counting from both your original classes
    Provides chunk_line_positions, word_frequencies, noise_threshold calculation
    Breaks circular dependency by providing frequency data BEFORE filtering

Stage 3: AdaptiveFilter ✅

    Takes frequency data + clean lines → Intelligently filtered text
    Uses your exact tuned filtering logic (76-threshold, pattern detection, spaCy grammatical filtering)
    No redundant frequency counting - uses injected Stage 2 data
    Preserves all your selective rules and measurement detection

Stage 4: PatternAnalyzer ✅

    Template-first approach for dynamic title chunk detection
    Universal anchor selection algorithm (not domain-specific)
    Finds chunks that appear in every file with consistent positioning
    Tracks static vs dynamic chunks across files for auto-completion
    Replaces your sophisticated title_template() and clustering logic

Key Architectural Wins

    Circular dependency eliminated: Frequency analysis happens before filtering
    Redundancy removed: Single frequency analyzer replaces multiple implementations
    Clean dataclass design: Proper field declarations, no mixed __init__ paradigms
    Universal algorithms: No hardcoded domain values
    Template-first processing: Proper baseline for detecting dynamic chunks
